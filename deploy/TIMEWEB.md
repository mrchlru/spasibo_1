# Timeweb App Platform — пути и проверки

## Критично: поле «Путь к директории проекта»

В [настройках деплоя из Dockerfile](https://timeweb.cloud/docs/apps/deploying-with-dockerfile) это поле задаёт **и расположение Dockerfile, и контекст сборки**.

- **Оставьте поле пустым**, если `Dockerfile` лежит **в корне репозитория** (как в этом проекте). Так используется корневой `Dockerfile` с `node:20-alpine` + `python:3.12-slim-bookworm`.

Если указать, например, только `frontend`, контекст станет папкой `frontend/`, и платформа может подхватить **чужой** `Dockerfile` из зависимости (`frontend/node_modules/jsbarcode/Dockerfile`) с образом **`node:8`** — в логах появятся огромные слои (~215 МБ), долгая загрузка и сборка **не того** приложения.

**Что сделать:** в панели приложения → параметры сборки → **«Путь к директории проекта» = пусто** (или явно корень репозитория, если интерфейс это позволяет), затем перезапустите деплой.

## Контекст сборки Docker

- **Корень проекта (build context):** каталог репозитория, где лежит `Dockerfile` (например клон на сервере: `/home/.../MugleHRbotTopManagment`).
- **Файл образа:** `Dockerfile` в корне репозитория.

## Внутри контейнера (после сборки)

| Назначение | Путь |
|------------|------|
| Рабочая директория процесса | `/app/backend` |
| Код FastAPI | `/app/backend/` |
| Собранный фронт (Vite `dist`) | `/app/frontend/dist` |
| `index.html` | `/app/frontend/dist/index.html` |
| Статика JS/CSS | `/app/frontend/dist/assets/` |

Переменные по умолчанию в образе: `SERVE_SPA=true`, `STATIC_ROOT=/app/frontend/dist`, **`PORT=80`**, **`EXPOSE 80`**, **`UVICORN_HOST=0.0.0.0`**. Поддержка Timeweb рекомендует слушать **IPv4** (`0.0.0.0`), если приложение должно быть доступно с их балансировщика по IPv4; при **`[::]`** внешний IPv4-трафик может не доходить до процесса.

Если HTTP-проба внутри контейнера идёт на **`localhost`**, он может резолвиться в **`::1`**. Тогда при **`UVICORN_HOST=0.0.0.0`** такая проба не попадёт в сокет; в логах при этом возможны **`HEAD /` 200** с **`172.18.x.x`** (IPv4 с моста). В спорных случаях задайте в панели **`UVICORN_HOST=::`** (uvicorn на Linux с `--host ::` часто принимает и IPv4, и IPv6) или согласуйте с поддержкой тип пробы.

**Важно:** в [документации Timeweb](https://timeweb.cloud/docs/apps/deploying-with-dockerfile) порт для пробы связан с **`EXPOSE`**. Если в образе **`EXPOSE 8080`**, а в переменных **`PORT=80`**, процесс слушает **80**, а проверка может идти на **8080** → `unhealthy`. Поэтому здесь **и `EXPOSE`, и дефолт `PORT` = 80**.

## Health check (состояние сервиса)

В **образе нет** директивы Docker **`HEALTHCHECK`**: деплой на Timeweb не зависит от встроенной пробы контейнера. По [документации Timeweb](https://timeweb.cloud/docs/apps/healthcheck-path), если **путь проверки в панели не задан**, после деплоя выполняется только **базовая проверка** (контейнер запустился); готовность HTTP-приложения платформа при этом **не** проверяет.

- **Эндпойнты в приложении:** `GET /health` (и **`HEAD /health`** — **200** без тела), ответ JSON `{"status":"ok"}`.
- **Панель:** при необходимости периодической HTTP-проверки укажите путь **`/health`** (без домена). Если путь **пустой** — соответствует схеме «только базовая проверка»; эндпойнты остаются для ручной диагностики и внешнего мониторинга.

Если в логах видно **`HEAD /`** и ответ **405**, платформа или балансировщик бьёт в корень: приложение отвечает на **`HEAD /`** кодом **200** (без тела), чтобы проба не считалась ошибкой.

### Поведение бэкенда (миграции в фоне)

Сервер начинает слушать порт **до** завершения SQL-миграций: `GET /health` возвращает **200** сразу (как рекомендует [документация Timeweb](https://timeweb.cloud/docs/apps/healthcheck-path) — эндпойнт не должен ждать БД). REST-эндпойнты до готовности схемы отвечают **503** с текстом «Сервис запускается…». **`GET`/`HEAD` на `/health` и `/live`** — всегда **200** (liveness). **`GET /ready`** — **503**, пока миграции не прошли или была ошибка.

### «Сборка успешна», но в конце «Deploy failed» / health `unhealthy`

Если в логах есть **`Build succeeded`** и в **логах приложения** видно, что uvicorn поднялся (строка вида `Uvicorn running on http://[::]:80` или `http://0.0.0.0:80`), а деплой всё равно падает с **`Health status: unhealthy`** через ~1–2 минуты, причина **не** в предупреждениях Vite. Зависимости Python ставятся в **`/opt/venv`** (образ), а не в системный Python.

Что проверить:

1. **IPv4 vs IPv6 / `localhost`.** По умолчанию в образе **`UVICORN_HOST=0.0.0.0`** (рекомендация Timeweb для доступа по IPv4). Если деплой **`unhealthy`**, а в логах приложения есть **`GET /health`** с **`::1`**, но платформа считает сервис недоступным, попробуйте **`UVICORN_HOST=::`** в панели или уточните у поддержки, с какого адреса идёт проба.
2. **Проверка здоровья** — **HTTP** на тот же порт, что в **`EXPOSE`** / слушает uvicorn (здесь **80**), путь **`/health`**. Снаружи приложение по-прежнему доступно по **443** на edge Timeweb ([деплой из Dockerfile](https://timeweb.cloud/docs/apps/deploying-with-dockerfile)).
3. Миграции в фоне: **`/health`** и **`/live`** не ждут БД. Если в панели включён **HTTP-путь проверки**, убедитесь, что порт совпадает с **`EXPOSE`** / **`PORT`** и что **`UVICORN_HOST`** не ломает пробу на **`localhost`** (см. выше).
4. **`PORT`** в панели и **`EXPOSE`** в Dockerfile должны совпадать с тем, на чём слушает uvicorn (по умолчанию **80**).

## Порт и хост привязки (uvicorn)

- Слушает **`PORT`** (по умолчанию **80** для Timeweb) и **`UVICORN_HOST`** (по умолчанию **`0.0.0.0`** в образе — доступ по IPv4 с edge Timeweb).
- Если платформа подставляет свой `PORT`, оставьте как есть.
- Для переопределения привязки задайте **`UVICORN_HOST`** в панели (см. выше про **`::1`** и пробы).

## Переменные окружения (обязательно задать в сервисе)

Все секреты и строки подключения из вашего `.env` / продакшена: `DATABASE_URL`, `ADMIN_API_KEY`, Telegram, при необходимости `CORS_ORIGINS` (ваш публичный URL через запятую), SMTP, S3 и т.д.

### Почта (SMTP)

Переменные `SMTP_HOST`, `SMTP_PORT`, `SMTP_USERNAME`, `SMTP_PASSWORD`, `ADMIN_EMAILS`, при необходимости `WEB_APP_LOGIN_URL` — см. [TIMEWEB_EMAIL_SETUP.md](../TIMEWEB_EMAIL_SETUP.md) в корне репозитория.

**Особенность App Platform:** бэкенд в контейнере может **не устанавливать TCP-соединение** с `smtp.timeweb.ru:465` (в логах `SMTPConnectTimeoutError`). Те же настройки на **Vercel, Railway или VPS** часто работают — ограничение исходящей сети платформы, а не обязательно неверный пароль. Уточните у поддержки Timeweb исходящий SMTP из Apps или используйте отправку почты с хоста, где SMTP доступен.

**Redis не обязателен:** в переменных задайте **`REDIS_ENABLED=false`**, чтобы не пытаться подключаться к `localhost:6379` в контейнере (иначе в логах будут ошибки Redis, кеш недоступен).

**Лог SQL:** по умолчанию **`SQLALCHEMY_ECHO=false`**. Для отладки в панели можно временно выставить **`SQLALCHEMY_ECHO=true`**.

### Object Storage (S3) — Timeweb Cloud

Ключи и бакет — в панели [Хранилище S3](https://timeweb.cloud/my/storage). Дефолты в коде под Timeweb:

- `S3_ENDPOINT_URL=https://s3.twcstorage.ru`
- `S3_REGION=ru-1`
- Плюс `S3_BUCKET`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`; при публичном CDN укажите `S3_PUBLIC_BASE_URL`.

Другой провайдер (Yandex, AWS и т.д.) — переопределите endpoint и регион в переменных окружения.

## Логи деплоя пустые или не грузятся (только «…»)

Это часто **не связано с кодом** — сбой отображения в панели или другой источник логов.

1. **Вкладка «Логи приложения»** (runtime), а не только «Деплой» — там stdout/stderr **uvicorn** (ошибки импорта, БД, миграции). Ошибка «Ошибка» на деплое при этом может быть в **логах приложения**.
2. **Расширенный режим логов** — в [документации Timeweb](https://timeweb.cloud/docs/apps/upravlenie-apps-v-paneli) для App Platform указано: логи деплоя бывают в **стандартном** и **расширенном** режиме (DEBUG). Включите расширенный, если в интерфейсе есть переключатель.
3. **Скачивание** — если есть кнопка «Скачать» / экспорт логов, сохраните файл и откройте локально.
4. **Браузер** — другой браузер, режим инкогнито, отключение блокировщиков и VPN (иногда ломают стриминг логов).
5. **Поддержка Timeweb** — укажите **ID приложения** (на скриншоте, например **176811**), время деплоя и что «логи деплоя не отображаются». Попросите прислать текст лога деплоя или диагностику со стороны платформы.

Пока панель молчит, соберите образ **локально** (см. ниже) — вы увидите ту же ошибку сборки/старта, что и на платформе.

## Сборка локально

```bash
docker build -t hr-spasibo:latest .
docker run --env-file .env -p 80:80 hr-spasibo:latest
```

Откройте `http://localhost:80/` — SPA; `http://localhost:80/health` — проверка. Чтобы не занимать порт 80 локально: `-e PORT=8080 -p 8080:8080`.
