# Timeweb App Platform — пути и проверки

## Критично: поле «Путь к директории проекта»

В [настройках деплоя из Dockerfile](https://timeweb.cloud/docs/apps/deploying-with-dockerfile) это поле задаёт **и расположение Dockerfile, и контекст сборки**.

- **Оставьте поле пустым**, если `Dockerfile` лежит **в корне репозитория** (как в этом проекте). Так используется корневой `Dockerfile` с `node:20-alpine` + `python:3.12-slim-bookworm`.

Если указать, например, только `frontend`, контекст станет папкой `frontend/`, и платформа может подхватить **чужой** `Dockerfile` из зависимости (`frontend/node_modules/jsbarcode/Dockerfile`) с образом **`node:8`** — в логах появятся огромные слои (~215 МБ), долгая загрузка и сборка **не того** приложения.

**Что сделать:** в панели приложения → параметры сборки → **«Путь к директории проекта» = пусто** (или явно корень репозитория, если интерфейс это позволяет), затем перезапустите деплой.

## Контекст сборки Docker

- **Корень проекта (build context):** каталог репозитория, где лежит `Dockerfile` (например клон на сервере: `/home/.../MugleHRbotTopManagment`).
- **Файл образа:** `Dockerfile` в корне репозитория.

## Внутри контейнера (после сборки)

| Назначение | Путь |
|------------|------|
| Рабочая директория процесса | `/app/backend` |
| Код FastAPI | `/app/backend/` |
| Собранный фронт (Vite `dist`) | `/app/frontend/dist` |
| `index.html` | `/app/frontend/dist/index.html` |
| Статика JS/CSS | `/app/frontend/dist/assets/` |

Переменные по умолчанию в образе: `SERVE_SPA=true`, `STATIC_ROOT=/app/frontend/dist`, `PORT=8080`.

## Health check (состояние сервиса)

- **HTTP:** `GET /health`
- **Ожидаемый ответ:** JSON `{"status":"ok"}` и код **200**.

В панели Timeweb укажите путь проверки: **`/health`** (без домена, только path).

### Поведение бэкенда (миграции в фоне)

Сервер начинает слушать порт **до** завершения SQL-миграций: `GET /health` возвращает **200** сразу (как рекомендует [документация Timeweb](https://timeweb.cloud/docs/apps/healthcheck-path) — эндпойнт не должен ждать БД). REST-эндпойнты до готовности схемы отвечают **503** с текстом «Сервис запускается…». При фатальной ошибке инициализации `/health` даёт **503**.

### «Сборка успешна», но в конце «Deploy failed» / health `unhealthy`

Если в логах есть **`Build succeeded`** и приложение пишет **`Uvicorn running on 0.0.0.0:8080`**, а деплой всё равно падает с **`Health status: unhealthy`** через ~1–2 минуты, причина **не** в предупреждениях Vite (большие чанки) и **не** в `pip install` от root — это только шум в логах.

Что проверить:

1. **Проверка здоровья** должна ходить на **HTTP** внутрь контейнера (порт из настроек, обычно **8080**), путь **`/health`**. Если платформа проверяет **HTTPS** к URL, где на edge некорректный SSL (в браузере бывает `ERR_SSL_PROTOCOL_ERROR`), проба может стабильно не проходить — уточните в поддержке Timeweb схему и таймауты healthcheck.
2. Если раньше деплой падал из‑за долгих миграций: в коде миграции вынесены в фон — проба `/health` не должна упираться в время применения SQL. Иначе уточните в поддержке **start period** / **интервал** или используйте **`HEALTHCHECK` в Dockerfile** ([документация](https://timeweb.cloud/docs/apps/healthcheck-path#docker-i-docker-compose)).
3. Убедитесь, что **`PORT`** в переменных совпадает с тем, на чём слушает процесс (в образе по умолчанию **8080**).

## Порт

- По умолчанию приложение слушает **`PORT`** (в образе **8080**).
- Если платформа подставляет свой `PORT` (часто 8080), оставьте как есть или задайте переменную окружения `PORT` в настройках приложения.

## Переменные окружения (обязательно задать в сервисе)

Все секреты и строки подключения из вашего `.env` / продакшена: `DATABASE_URL`, `ADMIN_API_KEY`, Telegram, при необходимости `CORS_ORIGINS` (ваш публичный URL через запятую), SMTP, S3 и т.д.

**Redis не обязателен:** без Redis приложение работает, кеш отключён.

### Object Storage (S3) — Timeweb Cloud

Ключи и бакет — в панели [Хранилище S3](https://timeweb.cloud/my/storage). Дефолты в коде под Timeweb:

- `S3_ENDPOINT_URL=https://s3.twcstorage.ru`
- `S3_REGION=ru-1`
- Плюс `S3_BUCKET`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`; при публичном CDN укажите `S3_PUBLIC_BASE_URL`.

Другой провайдер (Yandex, AWS и т.д.) — переопределите endpoint и регион в переменных окружения.

## Сборка локально

```bash
docker build -t hr-spasibo:latest .
docker run --env-file .env -p 8080:8080 hr-spasibo:latest
```

Откройте `http://localhost:8080/` — SPA; `http://localhost:8080/health` — проверка.
